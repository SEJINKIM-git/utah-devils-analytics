# Devils Insight AI 인수인계 문서

이 문서는 차후 운영 담당자가 비전공자여도 `Devils Insight AI`를 직접 업데이트하고 관리할 수 있도록 만든 실무용 안내서입니다.

## 1. 플랫폼 한 줄 설명

`Devils Insight AI`는 Utah Devils Baseball의 선수 등록, 시즌 기록 관리, 경기 리뷰, 라인업 시뮬레이션, 팀 분석, 선수 비교를 한 곳에서 운영하는 웹 플랫폼입니다.

## 2. 운영자가 가장 자주 하는 일

운영자는 보통 아래 5가지만 알면 됩니다.

1. 새 시즌 선수 명단 업로드
2. 경기 후 기록 파일 업로드
3. 대시보드에서 시즌별 반영 상태 확인
4. 잘못 올라간 파일 또는 선수 명단 정리
5. 필요할 때 AI(Codex, Claude 등)에게 수정 요청

## 3. 가장 쉬운 운영 흐름

### 3-1. 새 시즌 시작

1. `/upload` 또는 `/files`에서 선수 명단 파일 업로드
2. 대시보드에서 해당 시즌이 생겼는지 확인
3. 라인업, 선수 비교, 팀 분석에서 해당 시즌 선수들이 보이는지 확인

### 3-2. 경기 후 기록 반영

1. 경기 직후 공식 Excel 또는 Word 기록 파일 준비
2. `/upload?season=2026` 같은 형태로 현재 시즌 업로드 페이지 진입
3. 파일 업로드
4. 업로드 성공 메시지 확인
5. 대시보드, 선수 상세, 비교, 라인업, 팀 분석이 업데이트됐는지 확인

### 3-3. 잘못 업로드했을 때

1. 동일 시즌 누적 파일이면 수정된 최신 파일을 다시 업로드
2. 현장 경기 Excel/Word 파일이면 같은 날짜와 상대 경기 파일을 다시 업로드
3. 로스터 파일이면 `/files`에서 업로드 이력 확인 후 필요 시 재등록

## 4. 화면별 역할

### 4-1. 대시보드 `/`

- 시즌별 전체 타격/투수 기록을 보여줍니다.
- 검색창에서 선수 검색이 가능합니다.
- 선수 이름 클릭 시 현재 보고 있는 시즌을 유지한 채 선수 상세로 이동합니다.

### 4-2. 선수 비교 `/compare`

- 현재 선택한 시즌의 선수 명단만 비교 대상으로 보여줍니다.
- 시즌별 로스터 스냅샷과 실제 기록 데이터를 함께 사용합니다.

### 4-3. 라인업 `/lineup`

- 시즌별 선수 명단 기반으로 라인업을 구성합니다.
- 예상 타율, 출루율, OPS, 예상 득점이 계산됩니다.
- 상대 선발 ERA를 넣으면 예상 득점이 보정됩니다.

### 4-4. 일정 `/schedule` / 캘린더 `/calendar`

- 시즌 일정 및 수동 일정 관리용입니다.
- 날짜 클릭 후 제목, 시간, 장소, 메모를 직접 수정할 수 있습니다.

### 4-5. 팀 분석 `/team-analysis`

- 현재 시즌 팀 성적을 AI 분석합니다.
- 잠금 시즌은 공식 기록 업로드 전까지 비워 둡니다.

### 4-6. 경기 리뷰 `/game-review`

- 경기 단위 Word/Excel 파일을 바탕으로 리뷰를 저장합니다.
- 현재 연결 시즌 기준으로 과거 경기 목록을 가져옵니다.

### 4-7. 파일 관리 `/files`

- 선수 명단 업로드 이력을 확인합니다.
- 기존 명단을 재등록하거나 삭제할 수 있습니다.

## 5. 지원하는 공식 파일 형식

### 5-1. 로스터 파일

- 선수 명단 등록/수정용
- 대표 형식: `배번 | 이름`
- 업로드 시 선수 테이블과 시즌 로스터 스냅샷이 갱신됩니다.

### 5-2. 경기 기록 파일

- 경기 후 성적 업데이트용
- 구조화된 시즌 파일: `경기`, `타자`, `투수` 시트 지원
- 현장 경기 Excel: `타자 기록`, `투수 기록`, `주요 기록` 시트 지원
- 현장 경기 Word: 공식 경기 메모 `.docx` 지원

### 5-3. 공식 시즌 누적 파일

- `2022`, `2023`, `2024`, `2025` 같은 시트명
- `Spring/Fall/Summer` 상세 블록 시트
- 시즌 전체를 최신 파일 기준으로 교체합니다

### 5-4. 통산 파일

- `Career Totals` 지원

## 6. 시즌 데이터가 움직이는 방식

이 부분이 가장 중요합니다.

1. 시즌 대시보드는 선택한 `season` 기준으로 `batting_stats`, `pitching_stats`, `games`를 읽습니다.
2. 선수 상세 페이지도 이제 선택한 시즌 기준으로 표시됩니다.
3. 검색창과 선수 링크는 현재 보고 있는 시즌을 `?season=` 쿼리로 같이 넘깁니다.
4. `2026`은 공식 업로드 전까지 placeholder 시즌으로 잠겨 있을 수 있습니다.
5. 공식 2026 파일이 업로드되면 2026 시즌은 자동 활성화되고 이후 업로드 때마다 최신 기록 기준으로 갱신됩니다.

## 7. 2026 시즌이 자동 업데이트되는 원리

현재 시스템은 아래처럼 동작합니다.

1. 2026 경기 파일을 `/upload`에 업로드
2. 업로드 API가 해당 경기 또는 해당 시즌 데이터를 반영
3. 업로드 성공 시 관련 화면을 `revalidate`
4. 대시보드, 선수 비교, 선수 상세, 라인업, 팀 분석이 같은 시즌 문맥으로 다시 읽힘
5. 선수 상세 페이지는 `force-dynamic`이기 때문에 새로고침 시 최신 DB 상태가 반영됨

즉, 운영자가 따로 수식 계산을 하거나 페이지를 손으로 고칠 필요가 없습니다.

## 8. 비전공자용 최소 운영 체크리스트

파일을 올린 뒤 아래 6개만 확인하면 됩니다.

1. 업로드 성공 메시지가 떴는가
2. 업로드한 시즌 버튼이 대시보드에 보이는가
3. 대시보드 선수 수가 맞는가
4. 특정 선수 상세 페이지가 같은 시즌 기준으로 열리는가
5. 선수 비교 탭에 그 시즌 선수 명단이 맞게 뜨는가
6. 2026 시즌이면 새 경기 기록이 반영됐는가

## 9. 자주 수정하는 파일 위치

### 9-1. 화면

- `app/page.tsx`: 메인 대시보드
- `app/players/[id]/page.tsx`: 선수 상세
- `app/compare/page.tsx`: 선수 비교
- `app/lineup/page.tsx`: 라인업 페이지
- `app/components/LineupSimulator.tsx`: 라인업 계산
- `app/schedule/page.tsx`: 일정 페이지
- `app/calendar/page.tsx`: 캘린더
- `app/team-analysis/page.tsx`: 팀 분석
- `app/game-review/page.tsx`: 경기 리뷰
- `app/upload/page.tsx`: 업로드 화면
- `app/files/page.tsx`: 파일 관리

### 9-2. API

- `app/api/upload/route.ts`: 가장 중요한 업로드 로직
- `app/api/compare-data/route.ts`: 선수 비교 데이터
- `app/api/seasons/route.ts`: 시즌 목록
- `app/api/files/route.ts`: 파일 이력
- `app/api/game-review/route.ts`: 경기 리뷰 저장
- `app/api/analyze-team/route.ts`: 팀 분석

### 9-3. 공용 로직

- `lib/season.ts`: 시즌 선택 유틸
- `lib/seasonVisibility.ts`: 잠금 시즌/활성 시즌 관리
- `lib/rosterSnapshot.ts`: 시즌별 선수 명단 스냅샷 파서
- `lib/officialGameWorkbook.ts`: 공식 경기 Excel 파서
- `lib/parseDocxGameRecord.js`: Word 경기 기록 파서
- `lib/gameFileMeta.ts`: 파일명에서 날짜/상대/시즌 추출
- `lib/translations.ts`: 사이트 문구

## 10. 로컬 실행 방법

### 10-1. 최초 1회

```bash
npm install
```

### 10-2. 개발 서버 실행

```bash
npm run dev
```

### 10-3. 타입 체크

```bash
npx tsc --noEmit
```

### 10-4. 배포 전 빌드 확인

```bash
npm run build
```

참고:

- 현재 환경에 따라 `eslint-config-next` 경고가 날 수 있습니다.
- 실질적으로 중요한 건 `TypeScript` 오류와 `build` 실패 여부입니다.

## 11. 환경 변수

NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=

## 12. 배포 방식

이 프로젝트는 Vercel 배포 기준입니다.

1. 로컬에서 수정
2. `git commit`
3. `git push origin main`
4. Vercel이 자동 배포

배포 설정은 `vercel.json`에 있습니다.

## 13. 문제 발생 시 먼저 볼 것

### 13-1. 업로드는 됐는데 화면이 안 바뀔 때

1. 업로드 시즌이 맞는지 확인
2. 브라우저 강력 새로고침
3. 대시보드 URL에 `?season=2026` 같은 값이 맞는지 확인
4. `/upload` 성공 메시지와 `details.seasons` 확인
5. 필요 시 같은 파일을 다시 업로드

### 13-2. 특정 선수 기록이 이상할 때

1. 선수 상세 페이지 URL의 `season` 확인
2. 같은 시즌의 공식 파일을 다시 업로드
3. 선수 비교 탭에서 시즌별 명단이 맞는지 확인

### 13-3. 사이트가 에러 화면일 때

1. 최근 수정 파일 확인
2. `npm run build` 실행
3. `npx tsc --noEmit` 실행
4. 훅 순서 문제인지 확인
5. 최근 커밋 revert 대신 원인 수정 후 재배포 권장

## 14. Codex / Claude / 바이브 코딩 AI 활용 가이드

이 프로젝트는 차후 유지보수 시 AI를 적극 활용하는 것이 오히려 효율적입니다.

비전공자 운영자도 아래 방식으로 충분히 작업을 맡길 수 있습니다.

### 14-1. 언제 AI를 쓰면 좋은가

1. 문구 수정
2. 버튼/메뉴 추가
3. 업로드 형식 지원 확대
4. 특정 시즌/선수 데이터 표시 오류 수정
5. 모바일 UI 개선
6. 배포 오류 원인 찾기
7. 인수인계 이후 새로운 운영자 온보딩

### 14-2. 추천 모델

1. `Codex`
파일 수정, 커밋, push까지 이어서 맡기기 좋음

2. `Claude`
긴 문서 정리, 설명, 복잡한 로직 이해, 프롬프트 초안 작성에 좋음

둘 다 함께 써도 됩니다.

### 14-3. AI에게 요청할 때 가장 좋은 방식

아래 4가지를 같이 주면 결과가 가장 좋습니다.

1. 어떤 화면인지
2. 지금 문제인지 원하는 변경인지
3. 예시 파일 또는 스크린샷
4. 끝나면 `git commit`과 `git push`까지 하라고 명시

예시 프롬프트:

```text
선수 비교 탭에서 2026 시즌 로스터가 잘못 보입니다.
app/compare/page.tsx와 업로드 로직을 확인해서 시즌별 선수 명단이 정확히 보이게 수정해줘.
끝나면 타입 체크하고 git commit, git push까지 해줘.
```

```text
2026 경기 Word 파일 업로드 시 대시보드가 자동 갱신되게 해줘.
업로드 API와 선수 상세 페이지 시즌 표시도 같이 점검해줘.
비전공자도 이해할 수 있게 최종 설명은 한국어로 간단히 정리해줘.
```

### 14-4. AI에게 꼭 같이 요구하면 좋은 조건

1. 수정 후 `npx tsc --noEmit` 실행
2. 가능하면 `npm run build`도 확인
3. 어떤 파일을 바꿨는지 알려달라고 요청
4. 끝나면 `git commit`과 `git push`까지 하라고 요청
5. 설명은 한국어로, 짧고 실무적으로 정리해달라고 요청

### 14-5. AI를 쓸 때 주의할 점

1. “예쁘게 바꿔줘”보다 “어느 화면의 어떤 문제를 어떻게 바꿔달라”가 더 정확합니다.
2. 업로드/통계/시즌 로직은 반드시 실제 파일명과 시즌을 같이 알려주는 게 좋습니다.
3. 수정 후 production에서 실제로 새로고침까지 확인해야 합니다.
4. DB를 지우는 작업은 항상 조심해야 합니다.
5. 이유를 모르고 여러 파일을 한 번에 크게 바꾸기보다, 한 기능 단위로 요청하는 게 안전합니다.

## 15. 추천 운영 원칙

1. 공식 파일은 원본을 보관하고 업로드용 복사본을 따로 두기
2. 시즌별 파일명에 연도와 상대팀, 날짜를 명확히 넣기
3. 경기 후 바로 업로드해서 누락을 줄이기
4. 수정 작업은 작은 단위로 하고 즉시 배포 확인하기
5. AI에게 맡길 때는 “무엇을 바꾸고 왜 바꾸는지”를 분명히 말하기

## 16. 마지막 정리

운영자가 꼭 기억할 핵심은 세 가지입니다.

1. `업로드가 곧 업데이트`입니다.
2. `시즌 쿼리(?season=...)가 화면 문맥을 결정`합니다.
3. `Codex / Claude 같은 AI를 적극 활용해도 괜찮고, 오히려 권장`합니다.

이 프로젝트는 사람이 수동으로 모든 코드를 기억하는 방식보다, 운영 흐름을 이해하고 AI를 적절히 활용하는 방식이 훨씬 효율적입니다.
